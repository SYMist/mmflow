# 002. 모델 v3 정의 회의

- **목적**: 유저스토리 기반 v3 모델 확정, 폐기/재사용 범위 합의, 작업 분할 검토
- **참조**: `docs/v3-development-plan.md`, `meetings/001-kickoff-2026-04-08.md`

## 발언록

**PO**: 시작합니다. 지난 킥오프 이후 사용자가 직접 유저스토리를 정리해주셨고, 그 결과 모델 자체가 v2에서 크게 바뀌었습니다. `docs/v3-development-plan.md`를 기준으로 오늘 정식 합의하고, 폐기/재사용 범위와 작업 우선순위를 확정합시다.

**프로덕트 디자이너**: 먼저 모델 변화의 본질부터 짚고 가야 합니다. v2는 income → destination이라는 단순 N:N 그래프였는데, v3는 (1) 노드 타입이 3종으로 늘었고 (2) 도착 노드가 1개의 부모만 가지는 1:1 강제, (3) 그룹과 영역이라는 메타 개념이 추가됐습니다. 이건 단순 기능 추가가 아니라 그래프 모델의 근본적 재정의입니다.

**PM**: 동의합니다. 코드 관점에서도 v2의 `'income' | 'destination'` 타입이나 엣지 amount/ratio 같은 핵심 데이터 구조가 모두 사라집니다. 기존 코드의 30~40%는 폐기, 60~70%는 재사용이 가능할 것 같습니다.

**UX/UI 디자이너**: 그래도 React Flow 골격, 인라인 편집, 다크모드, 디자인 토큰, 월 피커 같은 것들이 살아남으니 다행입니다. 개념이 바뀐 거지 기반은 그대로 가도 됩니다.

---

**PO**: 그러면 v3의 핵심 결정들을 하나씩 검토합시다. 첫 번째, 노드 3종 모델에 대해 이견이 있나요?

**프로덕트 디자이너**: 서브 노드의 정의가 명확해진 게 좋습니다. 처음에 "패스스루 계좌" 같은 걸로 오해했는데, 사용자가 "주식 계좌의 배당금" 같이 자체 발생 수익이라고 정리해주신 게 핵심입니다. 이건 가구 경제 모델링에서 꽤 중요한 케이스입니다 — 부동산 임대 수익, 적금 이자 등 다 같은 패턴이거든요.

**UX/UI 디자이너**: 시각적으로는 도착 노드 옆에 작은 형태로 붙는다고 했는데, 두 가지를 결정해야 합니다. 첫째, 서브 노드의 크기 — 도착 노드의 60~70% 정도면 적당할 것 같고. 둘째, 위치 — 항상 도착 노드의 오른쪽 위에 붙이는 식으로 고정할지, 사용자가 위치를 조정할 수 있게 할지.

**프로덕트 디자이너**: 위치는 고정이 낫습니다. 자율 배치를 허용하면 도착 이동 시 어떤 offset을 유지할지 애매해지고, 시각적 일관성이 떨어집니다. 항상 오른쪽 위 고정으로 가죠.

**PM**: 동의합니다. 1:1 종속이고 항상 같이 움직이는 구조라면 부모 노드의 자식 컴포넌트로 렌더링하는 게 가장 단순합니다.

---

**PO**: 두 번째, 1:1 강제 정책. 두 번째 부모 연결 시 자동 복제 + 그룹 제안. 사용자 인지 부담은 어떻게 처리하나요?

**UX/UI 디자이너**: 솔직히 이게 가장 걱정됩니다. 사용자가 "왜 갑자기 노드가 두 개로 나뉘지?"라고 느낄 수 있어요. 모달이 "마법"처럼 동작하는 건 위험합니다.

**프로덕트 디자이너**: 그럼 모달 메시지를 명시적으로 작성해야 합니다. 단순히 "그룹화하시겠습니까?"가 아니라, "도착 노드는 한 개의 출발 노드만 가질 수 있어 새 도착 노드를 자동으로 만들었습니다. 이름과 계좌가 같으니 그룹으로 묶을까요?" 같이 동작 이유와 결과를 설명해야 합니다.

**UX/UI 디자이너**: 좋습니다. 그리고 모달에서 "그룹화 안 함"을 선택하면 같은 이름/계좌의 두 도착 노드가 별개로 존재하게 되는데, 이게 사용자에게 혼란일 수 있습니다. 시각적으로 "동일 이름/계좌인데 분리됨"을 알 수 있는 신호가 있어야 합니다 — 예를 들어 두 노드 사이에 옅은 점선 가이드 같은 거.

**PM**: 그건 v3 초기에 넣지 말고 사용자 피드백 받고 추가합시다. 일단은 모달 메시지만 잘 쓰고, 그룹화 거부한 경우는 그냥 두 별개 노드로 두는 식으로 시작하죠.

**PO**: 좋습니다. 모달 메시지를 명시적으로 작성하는 걸로 하고, 가이드 시각 표시는 v3 이후 검토.

---

**PO**: 세 번째, 그룹 매칭 키. 이름 + 계좌가 정확히 일치해야 그룹이 가능한데, 정규화는 어디까지 할까요?

**프로덕트 디자이너**: 사용자 입장에서 "주식 계좌"와 "주식계좌"는 같은 의도인데 그룹이 안 되면 답답할 겁니다. 최소한 양 끝 공백 trim과 연속 공백 단일화는 해야 합니다.

**UX/UI 디자이너**: 대소문자도 무시해야 한다고 봅니다. "Toss"와 "toss"는 같은 계좌니까요.

**PM**: 다만 정규화를 너무 적극적으로 하면 사용자가 의도적으로 다르게 만든 케이스(예: "비상금A", "비상금B")까지 같이 묶일 위험이 있습니다.

**프로덕트 디자이너**: 그건 글자가 다르니 안 묶입니다. trim + 공백 정규화 + 대소문자 무시까지는 안전합니다. 그 이상(예: 한글 자모 분리, 유사어 처리)은 하지 않습니다.

**PO**: 합의. 매칭 정규화는 (1) 양끝 trim (2) 연속 공백 → 단일 공백 (3) 대소문자 무시 까지.

---

**PO**: 네 번째, 영역(Zone). 단일 줌으로 가기로 했는데, 영역이 화면을 어떻게 분할하나요?

**UX/UI 디자이너**: 점선 박스로 영역을 그리되, 세로로 분할하는 게 자연스럽습니다. 멤버 2명이면 좌우 2등분, 그 위/아래에 공유 영역. 사실 더 좋은 건 가운데 공유 영역, 좌우에 멤버 비공유 영역 배치입니다 — 데이터 흐름이 공유 → 비공유로 가는 경우가 많을 테니까요.

**프로덕트 디자이너**: 공유 영역을 가운데에 두는 거에 동의합니다. 다만 박스 크기는 고정이 아니라 노드 개수에 따라 자동 조정되어야 합니다. 어느 한쪽 영역에 노드가 많으면 그쪽이 넓어지는 식으로.

**UX/UI 디자이너**: 자동 조정을 어떻게 구현할지가 문제인데, 우선 v3 초기에는 화면을 1:2:1 비율(좌멤버 : 공유 : 우멤버)로 고정하고, 사용자가 노드 위치를 자유롭게 옮길 수 있게 하는 정도로 시작합시다. 박스 크기는 안에 있는 노드들을 감싸도록 자동 계산.

**PM**: 합의. 영역 박스는 멤버 영역 + 공유 영역의 위치는 고정 (가운데 공유, 좌우 멤버), 크기는 안의 노드를 감싸도록 자동 계산.

**프로덕트 디자이너**: 멤버가 3명 이상인 경우는요?

**UX/UI 디자이너**: 가족 케이스일 텐데, 3명이면 좌중우로 멤버 배치하고 그 위(또는 아래)에 공유. 4명이면 2x2 그리드 + 가운데 공유. 솔직히 3명 이상은 흔한 케이스가 아니니 v3 초기에는 2명 케이스를 우선 완성하고, 3명+ 케이스는 별도로 처리합시다.

**PM**: 좋습니다. v3 MVP는 멤버 1명 + 멤버 2명만 지원, 3명 이상은 폼에서 입력은 받되 UI는 후속 작업으로.

---

**PO**: 다섯 번째, 멤버 추가/삭제 불가 정책. 사용자가 처음에 잘못 만들면 어떻게 하죠?

**UX/UI 디자이너**: 그래서 캔버스 생성 폼이 매우 신중해야 합니다. "이 설정은 나중에 바꿀 수 없어요"라는 경고를 분명히 표시해야 하고, 생성 직전 확인 단계가 있는 게 좋습니다.

**프로덕트 디자이너**: 또는 "캔버스 복제" 기능을 제공해서, 잘못 만들었으면 새로 만들고 노드들을 옮길 수 있게 하는 방법도 있습니다. 다만 이건 v3 초기에 넣기엔 부담이 큽니다.

**PM**: v3 초기에는 (1) 생성 폼에 명시적 경고, (2) 캔버스 삭제 기능만 제공. 복제는 후속.

**PO**: 합의.

---

**PO**: 여섯 번째, amount 모델. 모든 노드 amount는 사용자 입력, 엣지에는 amount 없음, 분배 검증으로 경고. 이 부분 한 번 더 확인합시다.

**프로덕트 디자이너**: 검증 규칙이 명확합니다. 출발/서브 노드의 amount보다 직접 자식 도착들의 amount 합이 크면 적색 경고. 그룹의 경우는 그룹 합이 어떻게 표시되나요?

**PM**: 그룹은 단순 합산만 표시합니다. 사용자가 별도 목표 금액을 두지 않기로 했으니 비교 대상이 없어서 경고는 안 나옵니다. 단순히 "이 그룹에 총 얼마가 모였다"는 정보만.

**UX/UI 디자이너**: 경고 표시 디자인은 어떻게 할까요? 적색 보더만으로는 약합니다.

**프로덕트 디자이너**: 노드 우상단에 작은 경고 아이콘(⚠) + 적색 보더 + 호버 시 툴팁으로 "분배 금액이 출발 금액을 초과합니다" 같은 메시지. 색상은 현재 디자인 토큰의 `--color-danger`를 그대로 사용하면 됩니다.

**UX/UI 디자이너**: 경고가 그룹 단위로도 발생할 수 있나요? 예를 들어 그룹 합이 어떤 외부 기준을 넘는 경우?

**PM**: 그룹은 별도 목표가 없으니 그룹 자체에는 경고가 안 발생합니다. 그룹 멤버 각자의 부모 출발 노드에서 발생할 뿐.

**PO**: 합의. 경고는 출발/서브 노드 단위로만 발생.

---

**PO**: 일곱 번째, 캔버스 라이프사이클. 리스트 뷰가 신규로 들어가는데, 라우팅 도입이 필요합니다.

**PM**: React Router를 도입하는 게 깔끔합니다. 다만 v3 초기에는 의존성을 늘리지 말고 단순 view state(예: `currentView: 'list' | 'canvas'`)로 시작하고, Phase 1 서버 통합 시점에 React Router로 정식 전환하는 방법도 있습니다.

**UX/UI 디자이너**: URL이 캔버스 ID를 가지고 있어야 공유/북마크가 가능하니, 어차피 서버 단계에서는 라우터 필요합니다. 하지만 v3 단계에서 미리 도입하면 작업이 늘어나니 PM 의견대로 view state로 시작합시다.

**프로덕트 디자이너**: 동의합니다. v3는 모델 안정화가 우선, 라우팅은 서버 통합 시점.

**PO**: 합의. v3는 view state로, 서버 통합 시 라우터 도입.

---

**PO**: 여덟 번째, v2 데이터 처리. 사용자가 데이터 폐기에 동의했는데, localStorage에 남은 v2 키는 어떻게 합니까?

**PM**: 두 가지 방법이 있습니다. (1) 첫 진입 시 v2 키를 자동 삭제 (2) 그대로 두고 v3는 새 prefix만 사용. (1)이 깔끔하지만, 만약 사용자가 나중에 "이전 데이터 복원해주세요" 하면 곤란합니다.

**프로덕트 디자이너**: (2)로 가되, 설정 화면에 "이전 버전 데이터 정리" 버튼을 두면 어떨까요?

**UX/UI 디자이너**: v3 단계에서 설정 화면을 따로 만드는 건 부담입니다. 그냥 (2)로 두고, JSON export로 백업 가능하니 사용자 책임에 맡깁시다.

**PM**: 합의. v2 키는 그대로 두고 무시. 명시적 정리 기능 없음.

---

**PO**: 마지막으로, 작업 분할과 PR 전략. 개발 계획에 8개 페이즈, 5개 PR 권장이 있는데 적절한가요?

**PM**: PR 분할은 적절한데, PR1(기반 + 노드 띄우기까지)이 가장 큰 덩어리입니다. 이걸 PR1a(스토어/타입 정의), PR1b(캔버스 메타 + 단일 캔버스 노드 렌더)로 더 쪼개도 좋겠습니다.

**프로덕트 디자이너**: 동의합니다. 큰 PR은 리뷰 어려우니 잘게 쪼개는 게 안전합니다. 5개 → 6개 PR로.

**UX/UI 디자이너**: PR3(그룹), PR4(영역) 사이에 시각 디자인 검토 시점을 두면 좋겠습니다. 그룹이 들어간 시점에 한 번 점검하고, 영역까지 들어간 시점에 한 번 더.

**PM**: 좋습니다. PR2 머지 직후, PR3 머지 직후, PR4 머지 직후 각각 디자인 리뷰 미니 미팅을 잡읍시다.

**PO**: 합의. PR 분할은 6개로 조정, 디자인 리뷰는 각 주요 PR 직후.

## 결정사항

### 모델 v3 (확정)

| 항목 | 결정 |
|------|------|
| 노드 타입 | 출발 / 도착 / 서브 (3종) |
| 도착-부모 관계 | 1:1 강제 (위반 시 자동 복제 + 그룹 제안 모달) |
| 서브 노드 | 도착 노드의 1:1 자식, 우상단 고정 위치, 함께 이동 |
| 엣지 | amount/ratio 없음 (도착 amount가 곧 흐름) |
| 그룹 | 동일 (이름 + 계좌) 도착 노드만 묶음 가능, 1 그룹 멤버십 |
| 그룹 매칭 정규화 | trim + 연속 공백 단일화 + 대소문자 무시 |
| 그룹 합산 | 멤버 amount 합 자동 표시 (별도 목표 없음) |
| 영역 (멀티 멤버) | 점선 박스, 가운데 공유 / 좌우 멤버 |
| 영역 줌 | 단일 줌 (영역별 독립 줌은 보류) |
| 영역 크기 | 노드를 감싸도록 자동 계산 |
