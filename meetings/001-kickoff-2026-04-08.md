# 001. 킥오프 미팅 — 제품 방향 및 피쳐 아이데이션

- **일시**: 2026-04-08
- **참석**: PO, PM, UX/UI 디자이너, 프로덕트 디자이너
- **목적**: 현재 코드베이스 기반으로 제품 방향 정의, 피쳐 아이데이션, 우선순위 결정

---

## 발언록

**PO**: 시작하겠습니다. 현재 프로토타입은 노드-엣지 기반 자금 흐름 시각화, 시나리오/월별 저장, 다크모드까지 갖춰져 있습니다. 여기서 "가구 경제 플래닝 도구"로 가려면 무엇이 빠져있고, 사용자가 반복 방문할 이유를 만들려면 어떤 피쳐가 필요한지 논의합시다.

**프로덕트 디자이너**: 현재 가장 큰 구조적 빈틈은 실제 가계부 기능이 없다는 점입니다. 지금은 "계획(Plan)"만 있고 "실행(Track)"이 없어요. 사용자가 월초에 플로우를 설계하고, 월말에 실제 지출과 비교할 수 없습니다. 계획 vs 실적 비교가 들어가야 반복 방문 루프가 생깁니다.

**UX/UI 디자이너**: 동의합니다. 그런데 실적 입력 UX를 무겁게 만들면 안 됩니다. 가계부 앱이 실패하는 주된 이유가 입력 피로도예요. 플로우 캔버스 자체에서 "이번 달 실제 얼마 썼는지" 간단히 입력하는 정도로 시작하고, 자동 연동(오픈뱅킹 등)은 나중에 고려하는 게 맞습니다.

**PM**: 서버와 인증이 아직 없는 상태에서, 서버 구축 → 인증 → 클라우드 저장이 선행되어야 이후 기능들이 의미가 있습니다. localStorage 기반에서는 기기 간 동기화도 안 되고, 광고도 붙이기 어렵습니다. 인프라를 먼저 잡아야 한다고 봅니다.

**PO**: PM 의견이 맞지만, 인프라만 하면 사용자 가치가 늘지 않습니다. 병렬로 가죠. 서버/인증은 기반으로 깔되, 사용자 관점 피쳐도 동시에 정의합시다.

**UX/UI 디자이너**: 현재 UX에서 개선이 필요한 부분을 정리하면 세 가지입니다. 첫째, 온보딩/튜토리얼입니다. 처음 들어온 사용자가 이 도구를 어떻게 쓰는지 모릅니다. 빈 상태 페이지는 있지만, "월급 노드를 만들고 → 생활비 노드를 연결하세요" 같은 가이드 플로우가 없어요. 둘째, 대시보드 뷰입니다. 캔버스가 메인인데, 숫자 중심으로 한눈에 파악할 수 있는 대시보드(파이차트, 바차트)가 있어야 합니다. 캔버스는 설계용, 대시보드는 확인용. 셋째, 모바일 대응입니다. 반응형 CSS는 있지만, 모바일에서 노드를 조작하기가 사실상 불가능합니다. 모바일은 리스트 뷰 기반으로 별도 UX가 필요합니다.

**프로덕트 디자이너**: 대시보드는 찬성하지만, 초기엔 캔버스 하단 Summary 영역을 확장하는 방향이 낫습니다. 별도 페이지를 만들면 복잡도가 올라가요. 현재 BottomSummary에 차트를 추가하는 방식으로 시작합시다.

**PM**: 모바일은 리소스가 많이 듭니다. PWA 설치 정도만 지원하고, 모바일 전용 뷰는 Phase 2 이후로 미루는 게 현실적입니다.

**PO**: 동의합니다. 모바일 전용 뷰는 후순위로.

**PO**: BM 관점에서 정리하겠습니다. 애드센스 수익화를 하려면 세 가지가 필요합니다. 첫째, 사용자 기반입니다. 회원가입/로그인이 필수입니다. 둘째, 체류 시간입니다. 반복 방문과 페이지 내 체류가 길어야 광고 수익이 남습니다. 셋째, SEO/공유입니다. 검색 유입을 위한 랜딩페이지, 공유 기능이 필요합니다. 그래서 반복 방문을 만드는 "계획 vs 실적" 기능, 체류 시간을 늘리는 "대시보드/리포트", 유입을 만드는 "공유 가능한 템플릿"이 핵심이라고 봅니다.

**프로덕트 디자이너**: 템플릿은 좋은 아이디어입니다. "신혼부부 가계부", "1인 가구 절약형" 같은 프리셋 시나리오를 제공하면 온보딩도 해결되고 SEO 랜딩도 만들 수 있어요.

**UX/UI 디자이너**: 템플릿 갤러리를 랜딩페이지에 넣으면 비회원도 구경할 수 있고, "이 템플릿으로 시작하기" 클릭 시 회원가입으로 자연스럽게 전환됩니다.

**PM**: 다만 템플릿은 콘텐츠 기획이 필요하니 Phase 1에서는 기본 템플릿 2~3개만 하드코딩하고, 사용자 생성 템플릿 공유는 Phase 2로 넘기죠.

**PM**: 지금까지 나온 아이템을 정리하고 우선순위를 매기겠습니다.

**PO**: 기준을 세우죠. Phase 1은 "서비스 런칭 가능 상태", Phase 2는 "성장 기능"으로 나눕시다.

**프로덕트 디자이너**: Phase 1에서 서버, 인증, DB 이전은 기술 기반이니 순서대로 가야 하고, 랜딩과 템플릿은 프론트엔드에서 병렬로 진행 가능합니다. 광고 배치는 랜딩과 함께 설계하면 됩니다.

**UX/UI 디자이너**: 디자인 방향도 이 자리에서 잡아야 합니다. 현재의 클린/미니멀 톤을 유지하는 게 맞다고 봅니다. 금융 도구는 신뢰감이 중요해서 과도한 장식은 배제해야 합니다. 광고는 콘텐츠 흐름을 방해하지 않는 위치에 넣어야 합니다. 사이드바 하단, Summary 영역 사이 정도가 적절합니다.

**프로덕트 디자이너**: 인증 UX는 소셜 로그인 우선으로 가는 게 좋겠습니다. Google과 Kakao를 기본으로 하고 이메일은 보조로 두죠. 가입 허들을 최소화해야 합니다.

**PM**: 랜딩페이지 구조는 히어로 섹션, 기능 소개, 템플릿 갤러리, CTA(회원가입) 순서가 자연스럽습니다.

**PO**: Phase 2 항목도 우선순위를 정합시다.

**프로덕트 디자이너**: 계획 vs 실적이 1순위입니다. 이게 없으면 사용자가 월 1회 방문하고 끝이에요. 이게 있어야 월 2~4회 방문이 생깁니다.

**UX/UI 디자이너**: 대시보드/차트가 2순위입니다. 실적 데이터가 들어오면 시각화가 자연스럽게 따라옵니다. 알림/리마인더가 3순위고, 온보딩 튜토리얼이 4순위입니다. 템플릿이 온보딩을 일부 대체하기 때문에 튜토리얼은 조금 밀려도 됩니다.

**PM**: 비정기 수입/지출 모델링과 목표 달성률 시각화는 TODO에 이미 있는 항목인데, Phase 2 후반으로 넣겠습니다.

**PO**: Phase 3은 확장 기능입니다. 모바일 전용 뷰, 사용자 템플릿 공유, 오픈뱅킹 연동, 다중 가구원 협업, PWA/앱 래핑. 이건 Phase 2가 끝난 후 시장 반응을 보고 결정합시다.

**PM**: 정리하면, Phase 1의 6개 항목이 MVP 스코프입니다. 서버/인증/DB 이전이 기술 기반이고, 랜딩+템플릿+광고가 사용자 대면입니다.

**PO**: Phase 1의 6개 항목을 MVP 스코프로 확정합니다. 다음 회의에서는 Phase 1의 기술 아키텍처를 구체화합시다.

## 결정사항

### Phase 1: 런칭 MVP

| # | 피쳐 | 설명 |
|---|------|------|
| 1 | 서버 구축 + API 설계 | 모든 후속 기능의 기반 |
| 2 | 회원가입/로그인 | 소셜 로그인 (Google, Kakao) 우선 |
| 3 | 클라우드 저장 (DB 이전) | localStorage → 서버 DB, 기기 간 동기화 |
| 4 | 랜딩페이지 | 히어로 → 기능 소개 → 템플릿 갤러리 → CTA |
| 5 | 기본 템플릿 2~3개 | 온보딩 해결 + 진입 장벽 제거 |
| 6 | 애드센스 광고 배치 | 사이드바 하단, 대시보드 영역 |

### Phase 2: 제품 고도화

| # | 피쳐 | 설명 |
|---|------|------|
| 1 | 계획 vs 실적 비교 | 반복 방문 핵심 루프 |
| 2 | 대시보드/차트 | Summary 확장, 파이/바차트 |
| 3 | 알림/리마인더 | 월초 플래닝 유도, 리텐션 |
| 4 | 온보딩 튜토리얼 | 단계별 가이드 플로우 |
| 5 | 비정기 수입/지출 | 보너스, 연납 보험 등 |
| 6 | 목표 달성률 시각화 | 동기부여, 게이미피케이션 |

### Phase 3: 확장

| 피쳐 |
|------|
| 모바일 전용 뷰 (리스트 기반) |
| 사용자 생성 템플릿 공유/갤러리 |
| 오픈뱅킹 연동 (자동 실적 입력) |
| 다중 가구원 협업 |
| PWA / 앱 래핑 |

### 디자인 방향

- 톤앤매너: 클린/미니멀 유지 — 금융 도구 신뢰감 우선
- 광고 배치: 콘텐츠 흐름 비침해 위치 (사이드바 하단, Summary 사이)
- 랜딩: 히어로 → 기능 소개 → 템플릿 갤러리 → CTA
- 인증: 소셜 로그인 우선 (Google, Kakao), 이메일 보조
